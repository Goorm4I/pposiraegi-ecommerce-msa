## 모니터링 구성 방식

### Q: 모니터링은 어떻게 구성하셨나요?

**A**: SRE의 Golden Signals를 기반으로 4가지 핵심 지표를 모니터링했습니다.

### 1️⃣ Golden Signals 기반 모니터링

#### Latency (지연 시간)
- **메트릭**: ALB TargetResponseTime P95
- **목표**: 200ms 미만
- **현재**: 150ms
- **모니터링**: 5분 단위 평균값 추적
- **알림**: 응답 시간 2초 초과 시 즉시 알림

#### Traffic (트래픽)
- **메트릭**: 실시간 RPS (Requests Per Second)
- **모니터링**: 분당 요청 수 추적
- **용도**: 트래픽 스파이크 감지, 스케일링 트리거

#### Errors (오류율)
- **메트릭**: HTTP 5XX 에러율
- **목표**: 0.1% 미만
- **현재**: 0.05%
- **알림**: 5XX 에러 발생 시 즉시 알림
- **상세**: 2XX, 4XX, 5XX 별도 모니터링

#### Saturation (자원 사용률)
- **CPU**: 80% 미만 목표 (알림: 80% 초과)
  - **현재**: 65%
- **Memory**: 85% 미만 목표 (알림: 85% 초과)
  - **현재**: 70%
- **범위**: 4개 마이크로서비스 모두 모니터링

### 2️⃣ 포괄적인 알림 시스템 (14개 알람)

#### 서비스별 CPU 알람 (4개)
```
pposiraegi-api-gateway-high-cpu
pposiraegi-user-service-high-cpu
pposiraegi-product-service-high-cpu
pposiraegi-order-service-high-cpu
```
- **임계값**: 80%
- **기간**: 5분
- **액션**: 오토스케일링 트리거 연동 가능

#### 서비스별 Memory 알람 (4개)
```
pposiraegi-api-gateway-high-memory
pposiraegi-user-service-high-memory
pposiraegi-product-service-high-memory
pposiraegi-order-service-high-memory
```
- **임계값**: 85%
- **기간**: 5분
- **액션**: 메모리 누수 감지, 인스턴스 재시작

#### ALB 알람 (2개)
```
pposiraegi-alb-high-5xx
pposiraegi-alb-slow-response
```
- **5XX**: 서버 오류 감지
- **응답 시간**: 2초 초과 감지

#### 서비스 Health 체크 (4개)
- **조건**: Running Tasks = 0 감지
- **서비스**: 4개 마이크로서비스
- **액션**: 서비스 다운 감지, 자동 복구

### 3️⃣ Infrastructure as Code (IaC)

**장점**:
- ✅ 모든 모니터링 설정을 Terraform 코드로 관리
- ✅ 환경 복제가 쉬움 (개발/스테이징/프로덕션)
- ✅ 설정 변경 시 Git 히스토리 추적 가능
- ✅ 팀원 간 설정 공유 및 리뷰 용이

**실제 코드 예시**:
```hcl
resource "aws_cloudwatch_metric_alarm" "high_cpu_api_gateway" {
  alarm_name          = "pposiraegi-api-gateway-high-cpu"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = "1"
  metric_name         = "CPUUtilization"
  namespace           = "AWS/ECS"
  period              = "300"
  statistic           = "Average"
  threshold           = "80"
  alarm_description   = "This metric monitors ECS CPU utilization"
  alarm_actions       = [aws_sns_topic.alerts.arn]
}
```

### 5️⃣ 통합 대시보드

**CloudWatch Dashboard URL**:
```
https://ap-northeast-2.console.aws.amazon.com/cloudwatch/home?region=ap-northeast-2#dashboards:name=pposiraegi-production-health
```

**대시보드 구성**:
- **Row 1**: Golden Signals (가용성, 응답 시간, 트래픽, 오류율)

## CloudWatch 비용 절감 마이그레이션 계획

### 🎯 비용 절감 전략

현재 CloudWatch 비용을 더욱 최적화하여 월 $5.00에서 월 $2.00 수준으로 줄일 수 있습니다.

### 1️⃣ 로그 비용 절감 (가장 효과 큼)

#### 개선안 1: 로그 레벨 필터링
```yaml
적용 방법:
  - ERROR, WARN 레벨만 CloudWatch로 전송
  - INFO, DEBUG는 로컬 파일로 저장

예상 효과:
  - 로그 용량: 2GB → 0.5GB (75% 감소)
  - 인제스트: 0.5GB × $0.50 = $0.25/월
  - 보관: 0.5GB × $0.03 × 23 = $0.35/월
  - 절감액: $0.40/월
```

#### 개선안 2: S3로 아카이브
```yaml
적용 방법:
  - CloudWatch Logs: 7일 보관 (무료)
  - 7일 후: S3로 자동 전송
  - S3 비용: $0.025/GB/월

예상 효과:
  - CloudWatch 보관: 무료 (7일)
  - S3 보관: 2GB × $0.025 = $0.05/월
  - 절감액: $1.33/월 (vs 30일 CloudWatch)
```

### 2️⃣ 알람 최적화

#### 개선안: 복합 알람 사용
```yaml
적용 방법:
  - 서비스별 CPU/Memory 알람 병합
  - Composite Alarm (복합 알람) 사용
  - 8개 알람으로 통합

예상 효과:
  - 8개 × $0.10 = $0.80/월
  - 절감액: $0.60/월
```

### 📋 비용 절감 마이그레이션 계획서

#### 단계 1: 로그 레벨 필터링 (즉시 적용)
```yaml
작업 항목:
  - 애플리케이션 로그 레벨 설정 (ERROR, WARN)
  - CloudWatch Agent 설정 수정
  - 테스트 및 검증

예상 비용:
  - 월 비용: $5.78 → $5.38
```
